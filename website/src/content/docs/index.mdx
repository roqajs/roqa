## Why Rift?

Rift takes a fundamentally different approach to building reactive UIs. Instead of shipping a runtime that manages state and DOM updates, Rift **compiles away** the reactive primitives at build time, producing vanilla JavaScript that directly manipulates the DOM.

<CardGrid>
  <Card title="Compile-Time Reactivity" icon="rocket">
    Reactive primitives like `cell()`, `get()`, and `set()` are inlined at build time. No runtime library needed for reactivity.
  </Card>
  <Card title="No Virtual DOM" icon="lightning">
    Updates go directly to the DOM. No diffing, no reconciliation overhead, no wasted cycles comparing trees.
  </Card>
  <Card title="Web Components" icon="puzzle">
    Components compile to native custom elements. Use them anywhere—with any framework or none at all.
  </Card>
  <Card title="Tiny Output" icon="document">
    The compiled output is minimal vanilla JS. Your bundle contains only what your app actually uses.
  </Card>
</CardGrid>

## A Quick Look

Here's a simple counter component in Rift:

```jsx
import { defineComponent, cell, get, set } from 'rift-js';

function Counter() {
  const count = cell(0);

  return (
    <button onclick={() => set(count, get(count) + 1)}>
      Count: {get(count)}
    </button>
  );
}

defineComponent('my-counter', Counter);
```

This compiles to optimized vanilla JavaScript:

```js
import { defineComponent, delegate, template } from 'rift-js';

const $tmpl_1 = template('<button> </button>');

function Counter() {
  const count = { v: 0, e: [] };

  this.connected(() => {
    const button_1 = this.firstChild;
    const button_1_text = button_1.firstChild;

		button_1_text.nodeValue = 'Count: ' + count.v;
		count.ref_1 = button_1_text;

    button_1.__click = () => {
      count.v = count.v + 1;
      count.ref_1.nodeValue = 'Count: ' + count.v;
    };
  });
}

defineComponent('my-counter', Counter);
delegate(['click']);
```

Notice how:

- `cell(0)` becomes a simple object `{ v: 0, e: [] }`
- `get(count)` becomes `count.v`
- `set()` is inlined with the DOM update (`count.ref_1.nodeValue = ...`)
- The reactive binding is compiled away entirely
